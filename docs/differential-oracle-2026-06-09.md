# Differential correctness testing vs upstream PolyML (2026-06-09)

## Why

The foundation audit (`docs/foundation-audit-2026-06-08.md`) concluded that the
project's deepest risk is a *class* of latent bugs — code correct on the tested
workload but wrong on the next one — with no systematic detector. Every Isabelle
keystone fix turned out to be one of these. The permanent fix is a **ground-truth
oracle**: run the same SML through the real upstream PolyML and through our `poly
run`, and diff. Any difference is a faithfulness bug in our port.

## Infrastructure

- `tools/build-oracle.sh` — builds the real upstream PolyML out-of-tree
  (`/tmp/polybuild/poly`); vendor/polyml stays pristine. Default int config
  (FixedInt 63-bit, `Int.maxInt = 4611686018427387903`) **matches**
  `/tmp/basis_loaded`, so that checkpoint is the correct diff target.
- `tools/diff-oracle.sh [--dir <d>] <file.sml ...>` — runs each snippet through
  both, extracts `@@<label>=<value>` lines (the filter strips REPL prompts/banner),
  and reports any divergence. Exit 1 if any category diverges.
- `tools/diff-corpus/*.sml` — 402 edge-case snippets across 12 runtime-feature
  categories (FixedInt overflow, IntInf compare/bitwise/gcd/div, Real64 + Real32
  math, Word arithmetic shifts, strings, lists/arrays, exceptions/packet-identity,
  conversions, real↔int boundaries, bit-mixing, Math constants). Generated by a
  parallel agent workflow targeting boundaries/negatives/zero/empty/min/max.

Run: `tools/diff-oracle.sh --dir tools/diff-corpus`

## Findings (first run: 8/12 agreed → after fixes 11/12)

| # | Bug | Severity | Status |
|---|-----|----------|--------|
| 1 | `IntInf` div/mod/divMod **by zero halts the interpreter** (returned TAGGED(0) instead of raising `Div`; divMod's TAGGED(0) was then read as a (q,r) tuple → crash) | HIGH (crash) | **FIXED** (raise real `Div`, ex_id TAGGED(7)) |
| 2 | `Real.round`/`Real.toInt TO_NEAREST`/`Real32.round` used round-half-**away** instead of round-half-to-**even** (2.5→3 not 2, 0.5→1 not 0) | HIGH (wrong value, core op) | **FIXED** (`round_ties_even`) |
| 3 | `Real32.Math.pi`/`e` = 0.0 | (stale checkpoint) | **FIXED by rebuild** (Real32 RTS was stubbed when the Jun-7 basis was exported; rebuilt) |
| 4 | basis-compiled **`IntInf.andb`** mis-takes the word path for `(tagged-negative, boxed)` args → truncates to 0 (`andb(~1, 2^80)=0`) | MEDIUM (narrow) | **OPEN — characterized** |

### #4 detail (open, for a focused follow-up)
- `andb(~1, 2^80) = 0` (ours) vs `2^80` (upstream). **Argument-order dependent**:
  `andb(2^80, ~1)` is correct.
- `poly_and_arbitrary` itself is **correct** (unit test `arb_and_neg_tagged_boxed_repro`
  passes; num-bigint two's-complement `&` is right). `orb`/`xorb` with the same args
  are correct.
- The SML `IntInf.andb` guard `largeIntIsSmall i andalso largeIntIsSmall j` (= two
  `RunCall.isShort`) wrongly evaluates *true* for `(tagged, boxed)` inside the
  **basis-compiled** `andb`, taking `Word.andb(largeIntToWord i, largeIntToWord j)`
  where `largeIntToWord 2^80 = 0` → 0. `andbFn` (the RTS) is never called.
- A **fresh REPL compile** of the identical guard, and of `andb`'s exact body
  (`myandb`), are BOTH correct on ours and upstream. Only the basis-compiled `andb`
  is wrong, and only `andb` (not the structurally-identical `orb`/`xorb`).
- Conclusion: a **compiler/optimizer mis-compilation** of `IntInf.andb` during the
  basis build (our SML compiler running on our interpreter produced wrong bytecode
  for that one function). Deep and narrow; deserves its own session with bytecode
  disassembly of the checkpoint's `andb` code object.

A second parallel workflow added **18 more categories** (Substring, Char
predicates/escapes, Word8, LargeWord, Vector/VectorSlice, Array/ArraySlice,
`Real.fmt` all modes, Real classification, ListPair, deep List, Int radix
fmt/scan, Option/Bool/General, deep String, more overflow corners, Real

Result: **all 18 new categories agree with upstream — 0 new divergences.**
Independently re-verified by re-running the full corpus from the main loop.
The corpus is now **30 categories / ~1181 differential comparisons**, and the
runtime matches upstream PolyML on **all of them except the single known
`IntInf.andb` compiler bug** (#4 / task #72). Strong evidence the runtime is
faithful across the Basis Library surface that HOL4/Isabelle exercise.

Run the whole thing: `tools/diff-oracle.sh --dir tools/diff-corpus` (29/30 agree;
the lone `intinf.sml` divergence is the documented andb case).

## Compiler-stress sweep + andb/orb root-cause (third pass)

A third workflow generated COMPILER-OPTIMIZER-STRESS programs (recursion, closures,
pattern matching, exceptions, higher-order, let-poly, big-case jump tables,
mini-interpreters, numeric-mix) — 12 themes — to hunt the `andb` mis-compilation
*class*. Result: **complex programs run faithfully; the ONLY divergence is the same
IntInf.andb/orb class.** Across the entire surface (30 Basis categories + 12
compiler-stress themes ≈ 1300 comparisons) andb/orb is the **sole** divergence.

Root cause (finding #4, task #72), via a bytecode dumper (RunCall reflection) +
the fixed disassembler:
- `IntInf.andb` AND `orb` (not `xorb`) are mis-compiled in the basis. The optimizer
  specializes andb/orb using AND/OR algebra ("and-ing a short positive / or-ing a
  short negative yields a short result" — IntInf.sml comment); xorb lacks the algebra.
- Our **bytecode backend mis-lowers** the specialized conditional: the basis_andb
  bytecode checks `isShort(A)` then `JUMP8_TRUE` straight to the word path, leaving
  the required sign check (`GREATER_EQ_SIGNED`/`CELL_FLAGS`/`EQUAL_WORD`) as DEAD CODE.
  So a short operand in the special-case slot wrongly takes the word path →
  `Word.andb/orb` of low words → truncation. `andb(~1,2^80)=0`, `orb(5,2^80)=5`.
- It's the bytecode backend (not the interpreter, RTS, or codetree optimizer):
  upstream NATIVE andb is correct (codetree is fine); a fresh REPL/file/structure
  compile of the identical body is correct (the specialization only fires in the full
  basis-build inlining context); the disassembled bytecode itself routes short-A to the
  word path bypassing the sign check.

Fix is a deliberate choice (task #72): (A) deep fix in the bytecode-backend conditional
lowering — the right fix, fixes the class, focused multi-step task; (B) tracked
basis-workaround patch (andb/orb always RTS) — small + correct but masks the backend
bug, needs all checkpoints rebuilt, and is a faithfulness deviation.

Bonus fix found en route: `disasm.rs` mis-decoded `IS_TAGGED_LOCAL_B` (no operand →
misaligned everything after); fixed (commit cc92c6b) — it feeds the JIT/profiler.
