# DRAFT upstream bug report: stage-0 bootstrap compiler mis-specializes IntInf.andb/orb

Status: DRAFT — ready to file against polyml/polyml on GitHub after a sanity pass.
Found 2026-06-09 by differential testing during the polyml-rs project (a Rust
re-implementation of the PolyML bytecode interpreter that reproduces this
byte-for-byte). All evidence below reproduces with stock upstream PolyML —
no polyml-rs components involved.

---

**Title:** Bootstrap-stage compiler mis-compiles `IntInf.andb`/`orb` for
(short, boxed) operands (interpreted builds; basis compiled by `polyimport`)

**Summary.** When the basis is compiled directly by the checked-in stage-0
bootstrap image (`bootstrap/bootstrap64.txt`) — i.e. by `polyimport` rather
than by the stage-2+ self-compiled compiler — `IntInf.andb` and `IntInf.orb`
are compiled to an incorrect specialized form. With a short operand in the
first argument position and a boxed (big) operand in the second:

```sml
IntInf.andb (~1, IntInf.pow (2, 80));   (* returns 0;            expected 2^80     *)
IntInf.orb  ( 5, IntInf.pow (2, 80));   (* returns 5;            expected 2^80 + 5 *)
IntInf.andb (IntInf.pow (2, 80), ~1);   (* correct (order-dependent)               *)
IntInf.xorb ( 5, IntInf.pow (2, 80));   (* correct (xorb is not specialized)       *)
```

**Reproduction** (Linux x86-64, an interpreted build makes the wrong code
*run*, but the mis-compilation is produced by the stage-0 image itself):

```sh
./configure --enable-native-codegeneration=no && make
cd <src>   # the directory containing basis/ and bootstrap/
cat > /tmp/repro.sml <<'EOF'
val () = Bootstrap.use "basis/build.sml";
val () = print (IntInf.toString (IntInf.andb (~1, IntInf.pow (2, 80))) ^ "\n");
val () = print (IntInf.toString (IntInf.orb  ( 5, IntInf.pow (2, 80))) ^ "\n");
EOF
<build>/polyimport bootstrap/bootstrap64.txt -I . < /tmp/repro.sml
# prints 0 and 5; expected 1208925819614629174706176 and ...81
```

The **shipped `poly` is unaffected** because the full bootstrap chain
recompiles the basis with the stage-2+ self-compiled compiler, which emits
correct code for the same source. The defect is only observable when the
basis from the stage-0 compile is actually executed (e.g. tooling that runs
`polyimport <stage-0> < basis-build` directly, or downstream projects reusing
`bootstrap64.txt`).

**Analysis of the emitted bytecode.** `IntInf.andb`'s source guard is
`largeIntIsSmall i andalso largeIntIsSmall j` (IntInf.sml). The stage-0
compiler specializes `andb`/`orb` (not `xorb`) using the and-with-short-
non-negative / or-with-short-negative algebra, emitting a form that extracts
the boxed operand's low-order word via an RTS call. Disassembling the
compiled `andb` closure shows the specialized guard is mis-lowered for the
first operand: after `isTagged(i)` the code jumps directly to the word path,
leaving the emitted sign check (`isTagged` → `i >= 0` / cellFlags == byte)
unreachable, while the second operand's mirror guard is correct. So a short
*negative* first operand (andb) or short *positive* first operand (orb)
wrongly takes the `Word.andb/orb`-of-low-words path and truncates the boxed
operand. (Both the byte-identical code object and the wrong results are
reproduced independently by upstream's own C++ interpreter executing
`bootstrap64.txt`, so this is not an execution bug — the stage-0 image's
compiler generates this code.)

**Likely root.** `bootstrap64.txt` is a pre-built image generated by an
earlier PolyML, so the defect lives in the *generating* compiler version's
arbitrary-precision specialization (codetree → bytecode path); the current
mlsource appears to produce the correct simple form when self-compiled.
A regenerated bootstrap image would presumably fix it; alternatively the
specialization in the generator is worth a look for the dead-sign-check
asymmetry described above.
