# polyml-rs foundation soundness audit (2026-06-08)

_Read-only ~23-agent workflow audit; 32 findings, 14 adversarially verified._

## Strategic recommendation

Do RUNTIME HARDENING next, not HOL4 Datatype (#68) and not the Isabelle antiquotation frontier. The audit is decisive on this. Every bug in the brief's hunt-list was LATENT — correct on tested workloads, wrong on the next one — and the verification confirmed several NEW latent bugs of exactly that shape sitting in the foundation right now: PolyCompareArbitrary returns 'equal' for ALL bignum comparisons >= 2^62 (high), EXTINSTR_INDIRECT_CLOSURE_W reads the wrong field/base and leaks a stack slot (high), FIXED_ADD/SUB/MULT and REAL_TO_INT silently wrap/truncate instead of raising Overflow (medium), IntInf.andb/orb/xorb/gcd/lcm return 0 on bignums (medium/low), and the JIT GET_THREAD_ID singleton bypass (medium). These are cheap to fix (mostly S/M) and directly retire the failure mode that has cost this project weeks repeatedly.

Why NOT #68 first: the audit confirms HOL4 Datatype needs essentially ZERO new foundational runtime work — it is SML-layer checkpoint assembly through the 'arithmetic middle' (relation/arithmetic/numpair/numSimps), the same Script->Theory pattern already proven ~15 times. It is multi-session SML slog with low foundational value, and — critically — the verification REFUTED the one runtime dependency that was claimed to bound it (the frozen-clock timing stub does NOT cause the METIS Time exception). So #68 is a feature, not a foundation, and it piles more weight on the same untested base. Isabelle's antiquotation expander is a months-long compiler-pipeline PORT, explicitly out of scope for a hardening pass.

The deepest finding is meta: the flagship capabilities (HOL4 MESON/METIS/Pelletier, Isabelle kernel, arbitrary-int basis) have NO CI-runnable regression guard — all 25/27 integration tests are #[ignore]'d, the non-ignored ones silently early-return when /tmp checkpoints are absent, the entire /vendor tree is git-ignored, and there is no CI of any kind. So any of the hardening fixes above could itself silently regress the headline workloads. This is why the recommended plan pairs the fixes with a regression fence.

Concrete next-session plan (1-2 sessions):
1. Fix the two HIGH bugs first (PolyCompareArbitrary boxed compare; EXTINSTR_INDIRECT_CLOSURE_W) — both S, both add a focused unit test that also fences the bug class.
2. Fix the Overflow family (FIXED_* and REAL_TO_INT/FLOAT_TO_INT) — M; share one correct EXC_overflow packet builder (the identity gotcha is the only subtle part) and apply to both interpreter and JIT.
3. Fix the bignum bitwise/gcd/lcm boxed paths and the JIT GET_THREAD_ID singleton — all S.
4. Stand up the regression fence the audit calls the single highest-leverage investment: (a) check in (or LFS) bootstrap64.txt and add ONE always-on smoke test running the ~1.1M-step interpreter path + a 3-line arithmetic/bignum proof; (b) wire the EXISTING-but-unused differential tester (polyml-jit/src/differential.rs) to the coverage_bootstrap corpus walk so interp-vs-JIT divergences surface in minutes instead of per-install bisection; (c) add the missing RESET-vs-RESET_R divergence unit test and a GC stress test promoting audit_no_residual_from_space_ptrs to an assertion. Without (b)/(c) the JIT GC-root gap and any future RESET/operand-desync refactor ship green.

Net: a focused hardening + fence pass retires the recurring latent-bug class, costs ~1-2 sessions of small fixes, and de-risks BOTH #68 and the Isabelle frontier for whenever they are pursued. Defer #68 to after the fence exists; defer the Isabelle antiquotation port indefinitely (it is a separate large project, not foundational hardening).

## Hardening backlog

Items marked DONE were applied in the 2026-06-08 hardening pass (commits 5ee457e, 7584f26).

### [1] ✅ DONE (high/S) crates/polyml-runtime/src/rts.rs:1629-1648 (poly_compare_arbitrary), test at rts.rs:3165-3173
- **Bug:** PolyCompareArbitrary is broken for boxed bignums. The reachable path (one operand >= 2^62) falls through to a constant PolyWord::tagged(0) (always-equal), so IntInf/LargeInt < > <= >= on bignums all misbehave: '<'/'>' return false, '<='/'>=' return true. The compiler only calls this RTS fn when at least one operand is boxed (isShort short-circuit handles two-tagged inline), so the dead both-tagged branch (which is ALSO sign-inverted: cmp(arg2,arg1)) masks the real defect. The arb_compare unit test enshrines the wrong (5,7)==1 convention.
- **Fix:** Replace the tagged(0) fallback with a real boxed compare: read both operands via poly_word_to_bigint and return sign(a.cmp(&b)) as TAGGED(-1/0/1); also fix the tagged branch to sign(arg1-arg2) (untag(arg1).cmp(&untag(arg2))). Keep the arg1.0==arg2.0 fast equality shortcut. Flip the arb_compare test to (5,7)=>-1, (7,5)=>1 and add a boxed case (2^70 vs 2^70+1 => -1). Upstream: arb.cpp:1858-1862 PolyCompareArbitrary=compareLong(arg2,arg1)=sign(arg1-arg2); arb.cpp:1253-1309. SML wiring: InitialBasis.ML:68/76-79, Int.sml:84-96.

### [2] ✅ DONE (medium/S) crates/polyml-runtime/src/rts.rs:1651-1681 (poly_or/and/xor_arbitrary)
- **Bug:** IntInf.andb/orb/xorb on boxed bignums silently return 0. Each handler implements only the both-tagged fast path (correct but DEAD: the SML side in IntInf.sml:77-90 only invokes the RTS fn when an operand is boxed), then falls through to PolyWord::tagged(0). So 100% of real RTS invocations return 0.
- **Fix:** After the both-tagged fast path, read both via poly_word_to_bigint and apply BigInt &/|/^, then bigint_to_poly_word(ctx, ...). num-bigint's signed BitAnd/BitOr/BitXor are two's-complement, matching upstream logical_long. Wire the real ctx (drop the leading `_` on the &mut RtsContext param). Add boxed regression test e.g. andb(2^64|0xF, 0xFF)=0xF plus a negative-operand case. Upstream: arb.cpp:1311-1456 (logical_long + and/or/xor_longc).

### [3] ✅ DONE (interp; JIT documented-gap) (medium/M) crates/polyml-runtime/src/interpreter/mod.rs (fixed_add/sub/mult + raise_overflow); rts.rs make_overflow_exception
- **Bug:** INSTR_FIXED_ADD/SUB/MULT use wrapping_add/sub/mul and unconditionally PolyWord::tagged(r), truncating mod 2^62 with no Overflow. Default config sets Int=FixedInt, so Int.+/FixedInt.+ compile straight to these. e.g. valOf Int.maxInt + 1 returns a wrapped value instead of raising Overflow. Silent wrong result; also present in the JIT.
- **Fix (interp, committed 1d1d007):** fixed_add/sub/mult compute in i128, range-check MIN_TAGGED..=MAX_TAGGED, else raise_overflow. The packet has ex_id==TAGGED(5) (EXC_overflow, INITIALISE_.ML:512) and ex_name=='Overflow' so `handle Overflow` matches (verified by tests/overflow.rs — `handle Overflow`, not `handle _`). Upstream: bytecode.cpp:925-981.
- **JIT (translate.rs:1540, documented gap):** the JIT FIXED_* arms still WRAP on i63 overflow. Bailing-to-interp was measured to cost ~9pp coverage (74.7%→65.5%) and broke 10 JIT arithmetic tests; a correct fix needs an overflow check that signals the trampoline to raise mid-body (an extern "C" fn can't build/unwind an SML packet itself). Kept as a wrap with an in-code KNOWN LIMITATION note; the JIT is off the default interpreter path that runs HOL4/Isabelle.

### [4] ✅ DONE (medium/M) crates/polyml-runtime/src/interpreter/mod.rs (REAL_TO_INT, FLOAT_TO_INT, real_to_int_round)
- **Bug:** Real.floor/ceil/trunc/round (and Real32 variants) of a value whose integer part > 2^62 returns a silently-wrong tagged int instead of raising Overflow. real_to_int_round did `r as isize` (Rust saturating cast) and the handler tagged unconditionally; no range check. NOT a SEGV in our interpreter (saturating cast avoids C's UB) — purely silent wrong value.
- **Fix (committed 1d1d007):** real_to_int_round now returns Option<isize> — pre-round magnitude check (MAX_TAGGED*1.5), finite check, post-round MIN_TAGGED..=MAX_TAGGED check; None → raise_overflow at both call sites (same EXC_overflow packet identity as the FIXED_* fix). Verified by tests/overflow.rs (FLOOR=CAUGHT, NORM_FLOOR=3). Upstream: bytecode.cpp:2014-2059.

### [5] ✅ DONE (high/S) crates/polyml-runtime/src/interpreter/mod.rs:2470-2477 (EXTINSTR_INDIRECT_CLOSURE_W)
- **Bug:** Wrong on three counts vs upstream indirectClosureW: (a) reads closure from peek(operand) but the operand is the FIELD INDEX and the compiler's genLocal pushes the closure to TOP first, so it should peek(0); (b) hardcodes field *p.add(1) instead of *p.add(1+item); (c) peeks-then-pushes (net +1) instead of replace-in-place (net 0), leaking a stack slot that desyncs every later frame/return. Latent: only emitted when a closure has local addr>=256 OR captured field index>=256 (genIndirectClosure else-branch); large machine-generated code (Isabelle/Pure) can hit it.
- **Fix:** Rewrite to: let item = self.fetch_u16_le()? as usize; let closure_word = self.peek(0)?; let p = closure_word.as_ptr::<PolyWord>(); let val = unsafe { *p.add(1 + item) }; self.stack[self.sp] = val; Ok(Continue). Compare the correct in-place EXTINSTR_INDIRECT_W (mod.rs:2445). Add a regression that forces W-form emission (closure with >256 captures, read high-index field) — the whole path is currently untested. Also re-audit EXTINSTR_moveToMutClosureW (mod.rs:2462) and the container-W family for the same peek-vs-top / push-vs-in-place confusion. Upstream: bytecode.cpp:2310-2311; emitter IntCodeCons.ML:1644-1659.

### [6] ✅ DONE (medium/S) crates/polyml-runtime/src/jit_bridge.rs:244-263 (jit_dispatch_get_thread_id)
- **Bug:** Under --jit, GET_THREAD_ID allocates a FRESH 8-word thread object every call instead of consulting the interpreter's cached singleton (interp.thread_object). Thread.self() identity becomes unstable, so setLocal writes object A and the next getLocal reads a fresh empty B → thread-local state silently lost (Isabelle Context.put / the_generic_context, HOL4 thread-local caches). Reachable: 0xd9 is inlined into getLocal/setLocal/self() callers and is not on the install skip list. Narrow live trigger (only the OUTERMOST do_call JIT dispatch runs as JIT code with MAX_JIT_DEPTH=0; the recursive thread-local carriers likely emit CALL_LOCAL_B and get install-filtered).
- **Fix:** Route jit_dispatch_get_thread_id through the singleton: if interp.thread_object is Some(t) return t; else alloc via jit_alloc_space_mut (9 words, flags slot=tagged(2)/PFLAG_SYNCH), store into interp.thread_object, return it. jit_alloc_space_mut == self.alloc_space and thread_object is already GC-forwarded (mod.rs:604), so storing it there makes it a tracked root automatically. Factor a shared Interpreter accessor used by both INSTR_GET_THREAD_ID (mod.rs:3448) and the JIT path so they cannot diverge. Upstream/mirror: alloc_stub_thread_object mod.rs:3448-3469; Thread.sml:362-395.

### [7] ⬜ TODO (medium/M) crates/polyml-runtime/src/rts.rs:351-360 (real_unary_stubs) + 369-375 (real_binary_stubs)
- **Bug:** All 22 Real32 PolyRealF* math/rounding RTS functions are stubbed to tagged(0). In 64-bit mode Real32 is TAGGED (float bits in high 32, low bit set), so this is type-valid (no SEGV) but decodes to 0.0f. Every Real32.Math.{sin,sqrt,...}, floor/ceil/round/trunc, pow, atan2, rem, copySign, nextAfter returns 0.0f. Worse: Real32.Math.pi/e are computed at basis-load (Real.sml:790-791) and baked as 0.0f into every checkpoint. Real32 basic +,-,*,/ are unaffected (those are bytecode float opcodes). HOL4/Isabelle don't use Real32, so impact is currently low.
- **Fix:** Implement the f32 variants mirroring the f64 ones using the tagged-float scheme already correct in interpreter box_float/unbox_float (mod.rs:3072-3078): read arg via ((w.0 as isize) >> 32) as i32 -> f32::from_bits, compute in f32, return ((bits as u64) << 32) | 1. Match upstream edge cases for FLog/FLog10 and FRound if bit-exact parity wanted. CRITICAL: preserve each name's register() ordinal (the loop order IS the registration order) to avoid invalidating warm /tmp checkpoints (MEMORY.md rts-token-staleness), or rebuild all. Upstream: reals.cpp:445-598, tagged-float ABI reals.cpp:198-217.

### [8] ⬜ TODO (low/S) crates/polyml-runtime/src/rts.rs:586-588 (registration) + 1184-1187 (poly_process_env_error_name)
- **Bug:** OS.errorName is doubly broken: (1) registered Arity1 but declared rtsCallFull1 (CALL_FAST_RTS2, threadId+syserr) so rts_call raises RtsArityMismatch BEFORE the body runs; (2) the body returns tagged(0) where upstream always returns a boxed string. In our guarded interpreter the worst case is a controlled InterpError (RtsArityMismatch, or NotAClosure if arity fixed), NOT the SEGV the original finding predicted. Trigger is narrow: REPL printing a bare syserror value, or Posix.Error.errorName fallback. HOL4/Isabelle SysErr paths carry errorMsg (safe stub), not errorName.
- **Fix:** Two changes: (a) register Arity2 matching the rtsCallFull1 declaration and the correctly-done sibling ErrorMessage (rts.rs:485); (b) make the body return a real string — minimally alloc_poly_string(ctx, b"ERROR<n>") using errno from args[1].untag(), or port stringFromErrorCode; even alloc_empty_string(ctx) stops it being garbage. Upstream: process_env.cpp:238-265.

### [9] ✅ DONE (low/S) crates/polyml-runtime/src/rts.rs:1740-1797 (poly_gcd_arbitrary / poly_lcm_arbitrary)
- **Bug:** IntInf.gcd/lcm on boxed bignums silently return 0 (only an i64 tagged fast path; boxed operands and tagged-overflow results fall to tagged(0)). gcd returning 0 then makes lcm/normalisation divide by zero or produce wrong fractions. Same incomplete-vs-arb.cpp class as the and/or/xor finding, lower frequency.
- **Fix:** Add a bigint fallback: read both via poly_word_to_bigint; gcd via num_integer::Integer::gcd; lcm = (|a|/gcd)*|b| guarding gcd!=0; bigint_to_poly_word. Also box the tagged-overflow result instead of returning tagged(0). Upstream: arb.cpp:1864-1904.

### [10] ⬜ TODO (low/S) crates/polyml-runtime/src/interpreter/mod.rs:2425-2429 (EXTINSTR_ATOMIC_EXCH_ADD)
- **Bug:** Pops two slots and returns tagged(0); upstream atomicExchAdd pops ONE (the addend), peeks the object, returns the OLD word0, writes old+addend-1 back. Our handler underflows the stack by one slot, ignores the object, returns constant 0. Dead today (opcode 0x96 is Legacy; no live SML caller in the modern compiler) but wrong code that would corrupt the operand stack if ever executed.
- **Fix:** Match upstream: let addend = self.pop()?; let obj = self.peek(0)?; read word0 as old; write old+addend-1 (raw tag arithmetic); replace top with old. Guard non-pointer obj defensively. Upstream: bytecode.cpp:1483-1494.

### [11] ✅ DONE (low/S) crates/polyml-runtime/src/interpreter/mod.rs:2414-2424 (EXTINSTR_ATOMIC_RESET / unlockMutex)
- **Bug:** Always returns False; upstream returns True iff oldValue==TAGGED(1) (uncontended). Mutex.unlock therefore always takes the slow PolyThreadMutexUnlock path (a redundant reset). Correctness-safe single-threaded, perf/semantics only; would become a real bug under a future multi-threaded runtime.
- **Fix:** Read oldValue before resetting; push True (tagged(1)) when oldValue==tagged(1) else False; still write tagged(0) into word0. Upstream: bytecode.cpp:1534-1542.

### [12] ⬜ TODO (low/M) crates/polyml-runtime/src/rts.rs:395-408 (PolyTiming* family)
- **Bug:** PolyTimingGetNow/GetUser/GetReal/GetSystem/GetGC*/GetChild* all return constant tagged(0) — frozen clock at epoch. Type-safe (Time.time=LargeInt), no SEGV. CORRECTION to the original finding: this is NOT the cause of the METIS 'spurious Time exception' (a frozen clock disables mlib's never-expire time budget rather than tripping it; the cited mlibOmega Time.fromSeconds is dead test-harness code unreachable from METIS_TAC) and is NOT the Datatype #68 bound. It is a faithfulness gap only; Timer/profiling SML reports 0.

Upstream: timing.cpp:205-231/452/534, Make_arb_from_pair_scaled scaling. Separately, re-root-cause the actual METIS abort (likely heap/GC under cumulative proving, not the clock).

### [13] ⬜ TODO (low/M) crates/polyml-runtime/src/interpreter/mod.rs:1207-1220 (sole GC trigger) + crates/polyml-runtime/src/space.rs:97-106 (alloc panics on exhaustion)
- **Bug:** GC fires once-per-step at 80%; MemorySpace::alloc() panics (aborts process) on exhaustion with no collect-and-retry. A single allocation larger than the remaining headroom in one step (e.g. Array.array of ~320MB+ on the default 1.6GB heap, or much smaller on the 256MB JIT/profile heap) aborts where upstream would GC/grow/use a separate segment. Clean abort, not corruption; safe in practice on the default heap.
- **Fix:** Route allocate()/RTS alloc through a fallible helper that, on try_alloc()==None, invokes self.gc() once and retries before panicking. Note upstream is stronger still: it also GROWS the heap and uses separate segments for oversized objects (processes.cpp:978/989/1022), so retry-only does not fully match for objects exceeding total capacity — a fixed-capacity single buffer is a second structural divergence to keep in mind. Reference: processes.cpp FindAllocationSpace.
