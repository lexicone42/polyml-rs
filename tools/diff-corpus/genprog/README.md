# genprog/ — frozen regression subset of the whole-program fuzzer

These are **vetted, byte-identical** SML programs produced by the type-directed
whole-program generators in `tools/diff-corpus-gen/`. They are checked into git
as a standing regression fence: `tools/diff-oracle.sh --dir tools/diff-corpus`
recurses into this directory, and that command is run by
`tools/regression.sh full`.

## Layout

```
genprog/
  arith_control/      60 programs  (genprog_arith.sml,            seed 42)
  lists_hof/          60 programs  (genprog_lists_hof.sml,        seed 42)
  datatypes_rec_exn/  60 programs  (genprog_dre.sml,              seed 42)
  strings_closures/   60 programs  (genprog_strings_closures.sml, seed 42)
  gc_pressure/        60 programs  (genprog_gc_pressure.sml,      seed 42)
```

**300 programs total.** Each is self-contained, deterministic, and prints one
`@@<label>=<value>` line. Each was confirmed byte-identical between OUR
`poly run /tmp/basis_loaded` and BOTH upstream oracles (native + bytecode-interp)
before freezing. They carry a `(* generated by ... -- DO NOT EDIT *)` header.

## Why a *subset* (and not all 2800)

The generators emit large batches (600/600/600/600/400 = 2800 with seed 42).
Running thousands of `poly` pairs would make `regression.sh full` slow. 60 per
dimension is enough to exercise every feature combination while keeping the
differential pass fast.

## Regenerate / expand

The frozen set is the first-60-by-sorted-filename of each dimension's seed-42
batch. To regenerate identically (or grow it):

```sh
# 1. (re)generate the full batches — see tools/diff-corpus-gen/README.md
# 2. freeze the first N per dimension:
for d in arith_control lists_hof datatypes_rec_exn strings_closures gc_pressure; do
  dest="tools/diff-corpus/genprog/$d"; rm -rf "$dest"; mkdir -p "$dest"
  ls tools/diff-corpus-gen/out/$d/*.sml | sort | head -60 | xargs -I{} cp {} "$dest/"
done
```

To expand coverage, bump the `head -60` (and/or add a fresh `GENPROG_SEED`).
Always re-verify any new files before banking:

```sh
tools/diff-oracle.sh --dir tools/diff-corpus/genprog                              # vs native
POLY_UPSTREAM=/tmp/polybuild-interp/poly tools/diff-oracle.sh --dir .../genprog   # vs interp
```

## Note on known divergences

The only known `diff-oracle` divergences in the whole `tools/diff-corpus/` tree
are the two **stage-0 IntInf.andb/orb** files (`intinf.sml`,
`intinf_bitwise_order.sml`) — a latent UPSTREAM stage-0-compiler bug we reproduce
byte-for-byte (see `docs/differential-oracle-2026-06-09.md`). **None of the
genprog programs touch that path** (the generators deliberately never emit
`andb`/`orb` on big `IntInf`), so every frozen genprog program AGREES.
